# ПМООС-RAG v0.17.0 «Faster» — резюме изменений

**Дата:** 26.06.2026
**Базовая версия:** v0.16.0 «Modular»
**Состав поставки:** `V0.17.0/` (исходники) + `MGRAG V0.17.0.zip`
(внутренняя папка `V0.17.0`).

Релиз — **пакет оптимизаций скорости / VRAM / надёжности без изменения поведения
для пользователя**. Приоритет пользователя (качество ответов и стабильность)
соблюдён: рискованные для выдачи изменения сделаны opt-in.

---

## Что нового (12 изменений, низкий риск)

### Базовый пакет (6)
1. **FP16 на GPU** для эмбеддера и реранкера (`embedding.fp16`/`reranker.fp16`).
   ~2× скорость кодирования, вдвое меньше VRAM. На CPU автоматически выкл.
2. **Синглтон моделей на процесс** — модель грузится в VRAM один раз и
   переиспользуется между модулями и ререндерами Streamlit.
3. **Параллельное расширение запросов (М4)** + предварительный батч-прогрев
   кэша эмбеддингов запросов. Убраны N последовательных вызовов LLM до генерации
   ответов на 75 замечаний.
4. **Память экспертизы: чтение базы один раз за прогон** + предвычисление
   токенов (снята квадратичная нагрузка при росте до сотен проектов).
5. **BM25-токенайзер понимает нормативные коды** — «СанПиН 2.2.1/2.1.1.1200-03»,
   «ГОСТ 17.2.3.02-78» индексируются единым токеном (точный лексический матч).
6. **JSON-ремонт с повтором** (`ai.json_repair_retry`) + одно переиспользуемое
   sqlite-соединение LLM-кэша.

### Доработки по итогам адверсариального ревью (6)
7. Исправлен рассинхрон ключа реестра моделей на fallback-пути fp16 (модель
   теперь кладётся под тем же ключом, по которому ищется).
8. **`embedding.fp16` по умолчанию ВЫКЛ (opt-in)**: half-точность слегка меняет
   векторы. При включении ключ кэша эмбеддингов учитывает fp16; рекомендуется
   переиндексация. `reranker.fp16` оставлен включённым (безопасно — логиты
   считаются заново, нет рассинхрона с индексом).
9. `setup_models.py`: корректная выгрузка модели из реестра перед `empty_cache`
   (иначе синглтон удерживал VRAM при самопроверке).
10. Память: проактивная инвалидация кэша после записи (страховка от грубой
    гранулярности mtime на FAT/exFAT/сетевых ФС).
11. BM25: срез завершающих `-`/`/` в нормализации кода (симметрия запрос↔документ
    на обрезанных на границе чанка кодах); пропуск дорогого разбора для чанков
    без цифр.
12. `chat_json`: повтор не дублирует роль `assistant` (защита чередования ролей
    для anthropic/gemini).
13. По итогам кросс-LLM проверки: реестр эмбеддера хранит (модель, фактический
    fp16); при cache-hit повторный экземпляр восстанавливает реальную точность —
    исключает запись fp32-векторов под fp16-ключ кэша на opt-in fp16 + сбое
    `.half()` (дормантно при дефолте embedding.fp16=False).

## Проверка

Выполнено: `py_compile` всех изменённых файлов (в т.ч. из доставляемой копии);
импорт всех модулей без циклов; точечные смоук-тесты логики (BM25-коды и
симметрия, кэш памяти и инвалидация, ключи моделей и fp16, дедуп расширений,
sqlite-кэш) — все зелёные; адверсариальный мультиагентный ревью (28 агентов):
23 находки, 9 подтверждено (все low), исправлены/учтены.

**Кросс-LLM проверка (gavel):** дифф независимо отревьюен DeepSeek и Llama-3.3-70b
(Groq). Оба подтвердили: fp16 opt-in по умолчанию — верно; реранкер fp16 — ок;
синглтон `_MODELS` корректен; добавление fp16 в ключ кэша обязательно и сделано.
Единственная содержательная находка (рассинхрон точности кэша на opt-in fp16 +
сбое `.half()`) закрыта правкой №13.

**Обязательно на ПК с окружением (RTX 3070 Ti) перед боевым использованием:**
- Холодный прогон Streamlit (AppTest): нет дублей ключей виджетов и deprecations.
- Замеры до/после: М2 индексация и М4 на ~75 замечаниях; контроль, что качество
  выдачи М4 не просело (особенно если включить `embedding.fp16`).
