# СтройПроект (ранее ПМООС-RAG)

Локальная модульная система с ИИ для эколога-проектировщика: систематизация
проектной документации (ПД) по **Постановлению Правительства РФ №87**, поиск
ответов на замечания государственной экспертизы по разделу **ПМООС/ООС**,
корректировка раздела и выгрузка данных в **УПРЗА «Эколог» / ИНТЕГРАЛ**.

> **Версия: 0.11.1 «Modular».** Полностью переписана после нерабочей сборки
> pmoos-rag10. Каждый из 6 модулей запускается отдельно (CLI) и из общего
> интерфейса (Streamlit). Поддержка нескольких ИИ-провайдеров на выбор
> (DeepSeek / OpenAI GPT / Google Gemini / Anthropic Claude / Kimi (Moonshot) / Mistral / локальная Ollama)
> с автоматической сменой модели. Оптимизировано под ноутбук с RTX 3070 Ti (8 ГБ).
>

>    встречалась в каких разделах и проектах — копится на диске (NetworkX, без
>    сервера), растёт при приёме ответов. Запрос «в каких проектах была техника X»
>    в Модуле 3 (`--who-uses`, `--knowledge`) и в интерфейсе.

<a name="установка"></a>
## 2. Установка

**Требования:** Windows 10/11, Python 3.10–3.12, ~6–8 ГБ свободного места
(под модели), для GPU — драйверы NVIDIA. Видеокарта необязательна (см. [GPU](#gpu)).

```bat
:: 1. Распакуйте архив, откройте папку проекта в проводнике
:: 2. Дважды кликните install.bat (или запустите в командной строке)
install.bat
```

`install.bat` создаст окружение `.venv`, поставит **PyTorch (CUDA 12.4)**,
затем зависимости из `requirements.txt`, и проверит доступность GPU.

> **Медленный/нестабильный интернет.** Установщик устойчив к обрывам:
> увеличенные таймауты и повторы, а если индекс по умолчанию
> (`files.pythonhosted.org`) не отвечает — он **автоматически переключается на
> зеркало** и при необходимости ставит пакеты по одному (успешно скачанные
> сохраняются). Если что-то всё же не докачалось — просто запустите `install.bat`
> ещё раз, загрузка продолжится. PyTorch при этом ставится один раз и не качается
> повторно. (Сообщение «Failed to initialize NumPy» в старой версии означало, что
> из-за обрыва не успел установиться NumPy — теперь это исправлено.)

> **Для сканированных PDF (OCR)** дополнительно установите Tesseract OCR и
> русский языковой пакет (`rus`). Без него текстовые PDF всё равно работают.
>
> **Локальные модели (без облака):** установите [Ollama](https://ollama.com),
> затем, например, `ollama pull qwen2.5:7b-instruct`.

Linux/macOS: `bash install.sh`.

<a name="модули"></a>
## 4. Модули и запуск по отдельности

Каждый модуль — отдельный скрипт в `modules/`. Запуск через `run_module.bat`
(активирует окружение и выполняет команду):

### МОДУЛЬ 1 — систематизация ПД (ПП-87) и версии
```bat
run_module.bat modules\module1_inventory.py --project "ОПОЧКА-ДУБРОВКА 83-26С" --uploads C:\files --object-type линейный
run_module.bat modules\module1_inventory.py --project "ОПОЧКА-ДУБРОВКА 83-26С" --show
```
Распознаёт раздел каждого файла, строит карту разделов (что есть / чего не
хватает по ПП-87), определяет версии и хронологию. Контакты — в интерфейсе.

### МОДУЛЬ 2 — RAG-база (индексация)
```bat
run_module.bat modules\module2_index.py --project "X" --background     :: в фоне
run_module.bat modules\module2_index.py --project "X" --status         :: прогресс
run_module.bat modules\module2_index.py --project "X" --pause
run_module.bat modules\module2_index.py --project "X" --resume
```
Гибридный поиск (dense bge-m3 + BM25 + RRF + реранкер). База Qdrant — **отдельно
от приложения**, дедупликация по содержимому, стабильные ID чанков (повторная
индексация не плодит дубли).

### МОДУЛЬ 3 — граф связей и каскад изменений
```bat
run_module.bat modules\module3_graph.py --project "X" --html
run_module.bat modules\module3_graph.py --project "X" --changed POS,TKR
```
Показывает зависимости данных между разделами и **что нужно пересчитать** при
изменении (напр. ПОС → выбросы → рассеивание → СЗЗ).

### МОДУЛЬ 4 — ответы на замечания ПМООС (3 блока)
```bat
run_module.bat modules\module4_answers.py --project "X" --remarks "замечания оос 1-75.docx" --object-type линейный --block 1
run_module.bat modules\module4_answers.py --project "X" --list
run_module.bat modules\module4_answers.py --project "X" --accept 5
run_module.bat modules\module4_answers.py --project "X" --edit 7 --text "Уточнённый ответ..."
run_module.bat modules\module4_answers.py --project "X" --block 2
run_module.bat modules\module4_answers.py --project "X" --block 3
```
- **Блок 1** — найти ответ по каждому замечанию с источником (раздел/файл/стр.);
- **Блок 2** — проверить расчёты, ссылки, актуальность нормативов;
- **Блок 3** — финальная проверка раздела целиком.
Решение по каждому пункту принимает пользователь (принять/править/отклонить).

### МОДУЛЬ 5 — корректировка ПМООС (DOCX) и таблица ответов
```bat
run_module.bat modules\module5_export.py --project "X" --oos "Раздел ПМООС.docx"
```
Формирует откорректированный раздел ПМООС (`.docx`) и таблицу ответов на **все**
замечания со столбцами **«ОТВЕТ»** и **«ИСТОЧНИК»** (раздел/файл/страница) —
в `.docx` и `.xlsx`.

### МОДУЛЬ 6 — выгрузка для УПРЗА «Эколог» / ИНТЕГРАЛ
```bat
run_module.bat modules\module6_uprza.py --project "X"
```
Таблицы источников выбросов и загрязняющих веществ (с кодами ЗВ) + задание на
ввод для расчёта рассеивания и карт.

<a name="данные"></a>
## 6. Где хранятся данные

Всё — в каталоге данных **вне приложения** (по умолчанию `~/.pmoos-rag`,
можно сменить через `PMOOS_DATA_DIR`):
```
~/.pmoos-rag/
  ├─ projects.json              реестр проектов
  ├─ .env                       ключи API
  ├─ config.yaml                настройки (необязательно)
  ├─ qdrant/                    векторная база (общая, не переиндексируется)
  ├─ emb_cache/                 кэш эмбеддингов
  ├─ models/                    скачанные модели (bge-m3, reranker)
  └─ projects/<проект>/
       ├─ inventory.json        карта разделов
       ├─ versions.json         версии и даты
       ├─ graph.json            граф связей
       ├─ answers.json          ответы Блока 1 + статусы решений
       ├─ decisions.jsonl       журнал решений (для обучения в будущем)
       ├─ tmp_uploads/          ВРЕМЕННЫЕ файлы ПД (можно удалять)
       └─ out/                  готовые .docx/.xlsx/.csv/.html
```
**Файлы проекта не хранятся постоянно** — после индексации `tmp_uploads` можно
очистить кнопкой в М1; карта разделов и база остаются.

<a name="gpu"></a>
## 7. Оптимизация под GPU (RTX 3070 Ti, 8 ГБ)

- torch ставится из индекса **cu124** (`install.bat`); эмбеддер bge-m3 идёт на CUDA.
- `embedding.batch_size=16`, `max_length=1024` — баланс скорости и VRAM.
- При нехватке VRAM (одновременно Ollama + индексация): уменьшите `batch_size`,
  для Ollama держите `num_ctx` ≤ 8192, либо запускайте индексацию и генерацию
  последовательно. Реранкер и эмбеддер освобождают CUDA-кэш.
- **Без NVIDIA-GPU**: всё работает на CPU (медленнее). `install.bat` сам поставит
  CPU-сборку torch, если cu124 недоступен.

### Qdrant в режиме сервера (для больших баз) и Ollama
- По умолчанию Qdrant работает встроенно (файлы в `~/.pmoos-rag/qdrant`). Для
  200+ проектов и сотен тысяч чанков запустите сервер:
  `docker compose up -d`, затем в `config.yaml`:
  `qdrant: { mode: server, url: "http://localhost:6333" }`.
- **Локальные модели (privacy, без оплаты API)** через Ollama на RTX 3070 Ti:
  установите Ollama, выполните `ollama pull qwen2.5:7b-instruct` (рекомендуется),
  в настройках слева выберите провайдер **Ollama** — модель подхватится
  автоматически. Держите `num_ctx` ≤ 8192, чтобы не переполнить 8 ГБ VRAM, если
  одновременно идёт индексация (эмбеддер bge-m3 тоже использует GPU).

<a name="зрелость"></a>
## 8. Что готово полностью, а что — каркас

**Готово и протестировано:** систематизация ПД и карта разделов (М1), версии и
хронология, граф связей и каскад (М3), формирование откорректированного ПМООС и

ответов с провенансом, проверкой согласованности и нормативов (М4, блоки 1–3),
маршрутизация ИИ-провайдеров и ключей, фоновая индексация с паузой.

**Каркас / требует данных и обкатки:**
- **Качество ответов М4** зависит от провайдера и полноты индекса — проверяйте и
  правьте предложения (human-in-the-loop встроен намеренно).
- **М6/УПРЗА**: формат «Эколог» закрытый, поэтому выгрузка — это CSV + задание;
  геометрию источников и привязку значений к источникам заполняет инженер.
- **Обучение на 200 проектах** (автонаписание разделов с нуля): база заложена и
  уже работает в части накопления ответов — **память экспертизы** (v0.13.0)
  копит принятые пары «замечание→ответ» и подмешивает их как примеры в новые
  ответы. Полное автонаписание разделов «с нуля» — следующий этап (см. ниже).
- Имена моделей в конфиге — дефолтные, могут устареть; проверяйте у провайдера.

<a name="диагностика"></a>
## 9. Диагностика проблем

- **«CUDA недоступна»** — переустановите torch из cu124 (см. `install.bat`),
  проверьте драйверы NVIDIA. Для работы на CPU ничего делать не нужно.
- **«Ключ API не виден»** — задайте ключ в интерфейсе или в `~/.pmoos-rag/.env`;
  кнопка в настройках сразу подхватывает ключ.
- **Ollama не найдена** — запустите `ollama serve` и `ollama pull <model>`.
- **Долгая первая индексация** — модели bge-m3/reranker скачиваются один раз в
  `~/.pmoos-rag/models` (статус виден в настройках).
- **Замечания не извлеклись из файла** — проверьте, что это таблица/нумерованный
  список; модуль также умеет извлекать замечания через ИИ.
