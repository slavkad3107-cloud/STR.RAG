# СтройПроект v0.28.0 «Swift» — резюме изменений

**Дата:** 07.07.2026 · **База:** v0.27.0 «Verified» · **Архив:** `STR.RAG V0.28.0.zip`

Целевой релиз ОПТИМИЗАЦИИ. Метод: фокусная многоагентная охота по 5 «горячим»
осям (стоимость рендера UI, горячий путь М4, пропускная способность индексации,
оперативная память, холодный старт) — 27 находок с честными оценками потерь,
каждая проверена по коду перед внедрением. **Качество ответов не тронуто нигде:**
все правки либо кэшируют неизменные данные, либо убирают дубли работы; результаты
поиска/ответов бит-в-бит те же (подтверждено 19 юнит- и 14 смоук-тестами).

## Индексация (самое ощутимое)

1. **Параллельный OCR.** Tesseract — внешний процесс, но страницы шли строго
   по одной: 300-страничный скан = 12–27 минут при простое 60–70% ядер. Теперь
   двухфазно: рендер страниц последовательно (PyMuPDF не потокобезопасен), OCR —
   пулом потоков (до 4). Тот же PNG → тот же текст, порядок страниц сохранён,
   кэш OCR потокобезопасен. **Ожидаемо в 2–3 раза быстрее на сканах.**
2. **Побайтовые дубликаты не парсятся.** Один том в разных папках раньше платил
   полный парсинг+OCR (до десятков минут) и только потом отбрасывался дедупом.
   Теперь sha256 сырых байтов (миллисекунды) до парсинга.
3. **Чистые сканы не гоняются через pdfplumber** (парсер таблиц берёт текст из
   символов — на скане их нет) + освобождение page-кэша (RAM на больших PDF).
4. COUNT(*) кэша OCR — раз в 200 вставок вместо каждой.

## Отзывчивость интерфейса (каждый клик = rerun всех 6 вкладок)

5. **Ollama-пробы больше не морозят UI.** При провайдере ollama каждый клик
   давал до 4× живых HTTP-проб с таймаутом 6–8 с — многосекундный фриз. TTL-кэш 15 с.
6. **М1: карта версий** (regex-классификация всех файлов + запись versions.json)
   — теперь по кэшу, пересчёт только при изменении файлов/инвентаря.
7. **М1: скан папки загрузок** (rglob+stat каждого файла) — троттлинг 2 с.
8. **М3: граф знаний** (полный json+DiGraph всех проектов) — кэш по сигнатуре файла.
9. **М2: журнал индексации** читается хвостом (64 КБ), а не целиком (мегабайты),
   и только в аварийных ветках статуса.
10. **Подпись «GPU/CPU» больше не импортирует torch** (секунды + сотни МБ RAM на
    первом рендере) — лёгкая проба драйвера NVIDIA; вычисления по-прежнему через torch.
11. Файл замечаний в М4 — без полной копии в память на каждый rerun (memoryview).

## Горячий путь М4 (75 замечаний)

12. **Определение тома ООС реранкало 60 кандидатов ради top-1** — ~половина всего
    rerank-бюджета прогона на второстепенное поле. Пул сжат до 16
    (`retrieval.oos_candidates`).
13. **Каскад считался дважды** на каждое замечание (downstream + explain_cascade) — раз.
14. **config.yaml читался 75 раз** за прогон (few-shot память) — cfg передаётся сверху.
15. **Префильтр алиасов ЗВ**: 110 regex-проходов по десяткам КБ текста на каждое
    замечание ×2 — теперь дешёвая проверка подстроки перед regex (результат
    идентичен: regex не может совпасть без вхождения подстроки).

## Оперативная память

16. **Интернирование токенов BM25**: одинаковые словоформы в тысячах чанков делят
    один объект — **минус ~200–500 МБ** на проектах в 20–50 тыс. чанков (BM25
    бит-в-бит идентичен). Также интернируются file/section в payload'ах.
17. **corp.tokens освобождается** после сборки BM25 (больше нигде не нужен).
18. **Потоковый pickle** корпуса: load/dump без промежуточного блоба ~60–180 МБ.
19. Проверено и СНЯТО подозрение: тексты корпуса НЕ дублируются (texts[i] и
    payload['text'] — один объект и в RAM, и в pkl).

## Холодный старт

20. **Создание поискового движка больше не грузит модель 2,3 ГБ** — dim для
    VectorStore разрешается лениво (нужен только при создании коллекции).
21. Кэш эмбеддингов: одно sqlite-подключение на процесс (память few-shot
    открывала новое на каждое из 75 замечаний); синхронная запись last_used
    на пути чтения — раз в 60 с (нужна только для FIFO-очистки).
22. Батч `record_many` в памяти экспертизы: приёмка N ответов = один проход
    вместо N перечитываний всей базы.

## Проверка

19 юнит-тестов зелёные; 14 целевых смоук-проверок (двухфазный extract_pdf —
текст/таблицы/порядок на живом PDF, ленивый dim без загрузки модели, raw-sha
дедуп, log_tail, record_many, префильтр ЗВ — поведение идентично); AppTest хаб +
6 модульных приложений — 7/7 без исключений; кэш графа М3 проверен фактически
(второй rerun не пишет на диск).

## Отложено (осознанно)

Кэш content_signature по doc_sha (заметно только на reindex); lookahead для
латинских «no»/«co» (нужен golden-set замер); вынос сетевых функций из hub.py;
version-aware near-dup. Роадмап прежний: baseline на golden-set → A/B
семантического чанкинга → contextual retrieval.
