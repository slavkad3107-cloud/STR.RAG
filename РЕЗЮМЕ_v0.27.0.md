# СтройПроект v0.27.0 «Verified» — резюме изменений

**Дата:** 07.07.2026 · **База:** v0.26.0 «Fidelity» · **Архив:** `STR.RAG V0.27.0.zip`

Полная перепроверка кода, концепции и функционирования после большого пакета
v0.25–v0.26. Метод: многоагентный ревью по 7 срезам ПРЯМО по коду (каждое
предложение состязательно верифицировано по `file:line`; 27 находок, 20
подтверждено) + собственные краевые проверки. Все подтверждённые дефекты и
безопасные оптимизации исправлены. Концепция подтверждена как верная (RAG по
индексу без хранения исходников, human-in-the-loop, модульность М1–М6).

## Исправленные дефекты моих же правок v0.26 (регрессии поймали ревьюеры)

1. **Тихая потеря документа из индекса** (high) — цепочка: «⏹ Стоп» помечал error
   уже завершённый файл → при возобновлении его чанки удалялись, а дедуп по
   устаревшему `known_shas` объявлял файл «дубликатом» и НЕ переиндексировал.
   Исправлены все звенья + 2 регрессионных теста.
2. **Коды пыли 2907/2909 были перепутаны** (high) — «цементная пыль» получала
   неверный код. Теперь по перечню ЗВ: 2907 = >70% SiO2 (динас), 2908 = 70-20%
   (шамот, цемент), 2909 = до 20% (доломит).
3. **Нормировка dense в RRF топила семантические находки** — деление веса на
   число dense-списков опускало одиночную семантическую находку ниже ХУДШЕГО
   элемента BM25. Теперь подокументная нормировка (средний reciprocal-rank) +
   отсечка нулевых BM25-скоров.
4. **Гейт достоверности давал системные ложные «low»** — сверка шла по 5
   фрагментам, а модель видела 8; норматив, процитированный самим экспертом,
   считался «выдумкой». Сверка теперь по тем же 8 фрагментам + тексту замечания.
5. **OOM-защита не работала на CPU** — сообщение DefaultCPUAllocator не содержит
   «out of memory»; добавлены CPU-паттерны (реранкер + эмбеддер).
6. **«🩺 Проверить окружение» грузил модель 2.3 ГБ** — убран мёртвый параметр
   dim; заодно эмбеддер больше НЕ загружается при 100% попадании в дисковый кэш.
7. **Память few-shot теряла раздел** — после отмены ложного srcs[:3] у
   неатрибутированных ответов писалась пустая секция; фолбэк на найденное поиском.

## М5 (правки жёлтым) — точность вставки

8. **Якорь с границами по цифрам**: «4.1» больше не находится внутри «14.1»,
   «4.12» и дат «04.11.2025» (правка вставала не туда). Падежи распознаются
   («в разделЕ 5», «по пунктУ 2.3»). Общий поиск якоря для предпросмотра и записи.
9. **Порядок правок к одному якорю** — был обратный (LIFO), стал исходный.
10. **Паритет предпросмотра и записи**: битый том виден в предпросмотре
    (`⚠ том не читается`), запись пропускает его и продолжает остальные;
    предпросмотр конвертирует .doc так же, как запись.
11. Объединённые ячейки таблиц не сканируются повторно.

## Интерфейс (по срезу UI)

12. **Бесконечный rerun при замене API-ключа** — keyed-состояние очищается.
13. **Два блока контактов затирали друг друга** (разные схемы в один
    contacts.json) — дублирующий блок убран.
14. **Подтверждения опасных действий «переезжали» на другой проект** — сбос
    арм-флагов и скачанного файла замечаний при смене проекта.
15. **Вечное «⏳ идёт» после гибели процесса** — индикатор шагов проверяет
    возраст «пульса» (>120 с → ⚠).
16. **Автопересборка карты разделов на каждый rerun** при постороннем файле в
    tmp_uploads — листинг фильтруется теми же расширениями, что и инвентарь.
17. Файлы из uploader'ов не перезаписываются на диск на каждый rerun.
18. **М4 предупреждает о CPU** до запуска (честное ожидание по времени).
19. **🛑 Защита от потери базы**: «Переиндексировать заново» при частично
    очищенных исходниках показывает «в базе N докум., на диске только M —
    отсутствующие будут удалены».

## Надёжность/скорость (без влияния на качество)

20. Реранкер больше не отключается МОЛЧА при сбое загрузки (лог + подсказка).
21. BM25-кэш инвалидируется в момент ПЕРВОГО изменения индекса (покрывает
    сбой/kill) и при reindex; сигнал «Стоп» не теряется при неудачном kill;
    защита от двойного старта в окне запуска (pid=0 + свежий пульс).
22. `ЗВ`: пользовательский YAML теперь реально ПЕРЕКРЫВАЕТ встроенные алиасы.
23. Батч-приёмка `set_decisions`: «Принять ВСЕ» на 75 ответов — 1 чтение/запись
    вместо 75 полных перезаписей answers.json.
24. SQLite-кэш эмбеддингов: ключи батчами по 900 (лимит переменных SQLite).
25. BM25-ранжирование через C-argsort; O(n²)-перезапись state при возобновлении
    заторможена до раза в секунду; точки Qdrant строятся лениво батчами
    (сотни МБ пикового RAM на больших томах); дубли текстов не кодируются
    моделью повторно; конфиг в few-shot пути читается один раз.
